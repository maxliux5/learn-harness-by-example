# 01 为什么 Agent 需要 Harness

很多 Agent 项目一开始都像一个很顺手的 demo：把任务拼进 prompt，调用模型，打印答案。这个版本通常很快能让人兴奋，因为它在一次输入上确实“跑通了”。但工程问题从第二次运行开始出现：同样的问题为什么答案变了，工具到底有没有被调用，失败时应该改 prompt、改工具，还是改终止条件？

Agent Harness 的作用，是把一次不透明的 Agent 运行，整理成可以运行、可以观察、可以评测、可以回放的工程对象。本教程不会一上来引入大框架，而是用一个研究助手贯穿 10 章，把每个组成部分一点点搭出来。

## 学习目标

读完本章，你应该先建立三条判断标准：

- Agent demo 只能证明“这次有输出”，不能证明它稳定、可查、可迭代。
- Harness 不是替代模型能力，而是管理模型、工具、状态、Trace、Eval 和 Replay 的运行外壳。
- 从第一章开始保留运行证据，会让后续定位失败、比较版本和写评测变得容易很多。

请注意这里的关键词：不是“让 Agent 看起来更高级”，而是“让 Agent 的行为可以被工程团队讨论”。当一个回答出错时，团队需要看到输入、模型响应、工具证据和错误位置，而不是只盯着最终文本猜原因。

## 本章问题

假设我们要做一个研究助手，用户输入一个问题，Agent 输出简短结论和依据。最直接的写法可能是：

```python
def research(question):
    return model.complete(question)
```

这段代码很容易写，也很容易演示，但它没有留下任何运行过程。你无法回答这些问题：

- 模型实际看到的上下文是什么？
- 它有没有使用外部资料，还是直接凭内部知识回答？
- 如果调用了工具，工具参数和工具结果是什么？
- 如果这次结果比上次差，差异来自模型、prompt、工具，还是随机性？
- 改完以后如何证明“真的更好”，而不是只换了一个更顺眼的样例？

Harness 要解决的就是这些问题。它把“我调用了一次模型”升级为“我运行了一次可以检查的流程”。

## 拆解思路

先把一次 Agent 运行拆成几类事实：

| 事实 | 如果不保存会怎样 |
| --- | --- |
| 输入任务 | 后续无法复现同一个问题 |
| 模型请求和响应 | 不知道模型是在什么上下文下做判断 |
| 工具调用和工具结果 | 无法确认答案是否基于证据 |
| 错误和终止原因 | 失败时只能猜测哪里断了 |
| 最终答案 | 没有稳定输出就无法做 Eval 和 Replay |

本章示例只做一件小事：对比 naive 版本和 harnessed 版本。前者只返回答案，后者返回答案和最小 Trace。这个 Trace 还很粗糙，但它已经把运行过程从临时 print 变成了结构化数据。

这也是整套课程的主线：每一章都只补一个能力，但它们最后会合成一个最小可用的 harness。

## 运行示例

运行：

```bash
python3 examples/ch01_why_harness.py
```

你会看到两段输出：

- `naive_research(question)`：只打印一个结论字符串。
- `harnessed_research(question)`：返回一个 dict，里面有 `answer` 和 `trace`。

读代码时重点看 [`examples/ch01_why_harness.py`](../examples/ch01_why_harness.py) 里的 `harnessed_research`。它没有引入真正的模型，也没有调用工具，只是记录了两个事件：

- `run.started`：这次运行从哪个 question 开始。
- `model.completed`：这次模拟模型调用完成，并记录 provider 和 message 数量。

这两个事件已经足够说明一个差异：naive 版本只能让你看到“答了什么”，harnessed 版本开始让你看到“怎么运行的”。

## 观察与改进

运行后不要只看答案内容，请观察 JSON 结构：

- `answer` 是最终输出，未来可以给用户展示，也可以进入 Eval。
- `trace` 是事件列表，未来可以扩展为 JSONL、写入文件、进入 Replay。
- 每个事件都有 `timestamp`、`name` 和 `payload`，这是一条可继续演进的事件协议。

这个 harness 现在还能防住一个很常见的误判：只因为答案看起来合理，就以为系统可靠。示例中的 `naive_research` 也会给出合理答案，但它没有证据链。未来当答案变差时，你没有任何线索；而 `harnessed_research` 至少能告诉你这次运行的输入和模型完成事件。

可以尝试做两个小实验：

1. 改掉 `question`，观察 `trace[0]["payload"]` 是否跟着变化。
2. 给 `trace` 增加一个 `prompt.built` 事件，记录你准备发给模型的 message 数量。

这些实验的目的不是增加功能，而是训练一种习惯：只要某个信息会影响结果，就应该考虑是否进入 harness 的结构化记录。

## 常见误区

- 误区一：只要模型足够强，就不需要 harness。模型越强，行为越复杂，越需要可观察性。
- 误区二：先把 Agent 做完，再补 Trace 和 Eval。等流程复杂后再补，通常已经不知道哪些中间事实最关键。
- 误区三：Harness 等于某个商业平台。本教程讲的是工程结构，平台只是落地方式之一。
- 误区四：Trace 只是 debug log。Trace 是后续 Eval、Replay、对比和排错共同使用的数据基础。

## Checkpoint

本章结束时，你应该能用自己的话解释：

- 为什么 Agent demo 不等于 Agent 系统。
- Harness 管的是运行过程，而不是直接提升模型智商。
- 为什么一个最小 harness 也应该留下 answer 之外的运行证据。
